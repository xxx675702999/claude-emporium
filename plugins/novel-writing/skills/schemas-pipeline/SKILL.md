# Pipeline Artifact Schemas

## Overview

The InkOS pipeline generates intermediate artifacts during chapter creation. These artifacts are stored in `books/<id>/story/runtime/` and follow strict schemas for validation and debugging.

## Pipeline Stages

1. **Prepare** → `chapter-XXXX.intent.md` + `chapter-XXXX.context.json` + `chapter-XXXX.rule-stack.yaml`
2. **Write** → `chapter-XXXX.md` (includes inline length check)
3. **Settle** → `chapter-XXXX.delta.json`
4. **Persist** → Updates to truth files
5. **Audit** → `chapter-XXXX.audit.json`
6. **Revise** → Final `chapter-XXXX.md`

## 1. chapter-intent.json (chapter-XXXX.intent.md)

Generated by Preparer agent. Defines chapter goals, directives, and hook agenda.

### Schema

```typescript
{
  chapter: number,                    // Chapter number (≥1)
  goal: string,                       // Primary chapter objective (required)
  outlineNode?: string,               // Corresponding outline beat
  sceneDirective?: string,            // Scene-level guidance
  arcDirective?: string,              // Character arc guidance
  moodDirective?: string,             // Emotional tone guidance
  titleDirective?: string,            // Title generation hint
  mustKeep: string[],                 // Elements that must appear
  mustAvoid: string[],                // Elements to avoid
  styleEmphasis: string[],            // Style priorities for this chapter
  conflicts: Array<{
    type: string,                     // Conflict category (required)
    resolution: string,               // How it resolves (required)
    detail?: string                   // Additional context
  }>,
  hookAgenda: {
    pressureMap: Array<{
      hookId: string,                 // Hook identifier (required)
      type: string,                   // Hook type (required)
      movement: "quiet-hold" | "refresh" | "advance" | "partial-payoff" | "full-payoff",
      pressure: "low" | "medium" | "high" | "critical",
      payoffTiming?: "immediate" | "near-term" | "mid-arc" | "slow-burn" | "endgame",
      phase: "opening" | "middle" | "late",
      reason: "fresh-promise" | "building-debt" | "stale-promise" | "ripe-payoff" | "overdue-payoff" | "long-arc-hold",
      blockSiblingHooks: boolean      // Prevent other hooks in same family
    }>,
    mustAdvance: string[],            // Hooks that must progress this chapter
    eligibleResolve: string[],        // Hooks ready for payoff
    staleDebt: string[],              // Hooks overdue for advancement
    avoidNewHookFamilies: string[]    // Hook types to avoid introducing
  }
}
```

### Hook Movement Semantics

- `quiet-hold`: Hook exists but not mentioned this chapter
- `refresh`: Re-mention without advancing (keep pressure alive)
- `advance`: Move hook forward (new information/complication)
- `partial-payoff`: Deliver part of the promise
- `full-payoff`: Complete resolution

### Validation Rules

- `chapter` must be positive integer
- `goal` must be non-empty
- Each conflict must have `type` and `resolution`
- Hook pressure map entries must have valid enum values
- `mustAdvance` hooks should appear in `pressureMap`

## 2. context-package.json (chapter-XXXX.context.json)

Generated by Preparer agent. Contains relevance-scored context selection.

### Schema

```typescript
{
  chapter: number,                    // Chapter number (≥1)
  selectedContext: Array<{
    source: string,                   // Context source identifier (required)
    reason: string,                   // Why this context was selected (required)
    excerpt?: string                  // Optional excerpt for large sources
  }>
}
```

### Context Sources

Common source identifiers:
- `current_state`: Current state card
- `hooks:<hookId>`: Specific hook
- `summary:<chapterNum>`: Chapter summary
- `character:<name>`: Character profile
- `subplot:<id>`: Subplot thread
- `emotional_arc:<characterId>`: Emotional progression
- `volume_outline:<section>`: Outline segment

### Selection Strategy

Preparer uses relevance scoring based on:
1. Recency (recent chapters weighted higher)
2. Hook pressure (high-pressure hooks prioritized)
3. Character involvement (mentioned in intent)
4. Subplot activity (active subplots over dormant)
5. Outline alignment (matching current beat)

### Validation Rules

- `chapter` must be positive integer
- Each context entry must have non-empty `source` and `reason`
- Total context should fit within token budget (tracked separately)

## 3. rule-stack.yaml (chapter-XXXX.rule-stack.yaml)

Generated by Preparer agent. Compiles layered rules with override edges.

### Schema

```typescript
{
  layers: Array<{
    id: string,                       // Layer identifier (required)
    name: string,                     // Human-readable name (required)
    precedence: number,               // Higher = higher priority
    scope: "global" | "book" | "arc" | "local"
  }>,
  sections: {
    hard: string[],                   // Hard constraints (must not violate)
    soft: string[],                   // Soft preferences (should follow)
    diagnostic: string[]              // Diagnostic checks (for audit)
  },
  overrideEdges: Array<{
    from: string,                     // Source layer (required)
    to: string,                       // Target layer (required)
    allowed: boolean,                 // Whether override is permitted
    scope: string                     // Override scope (required)
  }>,
  activeOverrides: Array<{
    from: string,                     // Overriding rule (required)
    to: string,                       // Overridden rule (required)
    target: string,                   // What is being overridden (required)
    reason: string                    // Justification (required)
  }>
}
```

### Layer Precedence

Standard layers (highest to lowest):
1. L4 - Current Focus (local, per-chapter)
2. L3 - Author Intent (arc-level)
3. L2 - Book Rules (book-level)
4. L1 - Genre Profile (global)

### Rule Sections

- **Hard**: Immutable constraints (e.g., "no time travel", "protagonist cannot die")
- **Soft**: Stylistic preferences (e.g., "prefer dialogue-driven scenes")
- **Diagnostic**: Audit triggers (e.g., "flag if chapter has no conflict")

### Override Semantics

When a higher-precedence layer overrides a lower layer:
- Override must be explicitly declared in `activeOverrides`
- Override must have a `reason` explaining why
- Override edge must exist in `overrideEdges` with `allowed: true`

### Validation Rules

- At least one layer must be present
- Layer `precedence` values should be unique
- Override edges must reference existing layers
- Active overrides must have corresponding allowed edges

## 4. delta.json (chapter-XXXX.delta.json)

Generated by writer agent during settlement phase. Contains state updates.

### Schema

```typescript
{
  chapter: number,                    // Chapter number (≥1)
  currentStatePatch?: {
    currentLocation?: string,
    protagonistState?: string,
    currentGoal?: string,
    currentConstraint?: string,
    currentAlliances?: string,
    currentConflict?: string
  },
  hookOps: {
    upsert: Array<HookRecord>,        // Create or update hooks
    mention: string[],                // Hook IDs mentioned (no change)
    resolve: string[],                // Hook IDs to mark resolved
    defer: string[]                   // Hook IDs to mark deferred
  },
  newHookCandidates: Array<{
    type: string,                     // Hook type (required)
    expectedPayoff: string,           // Expected resolution (default "")
    payoffTiming?: "immediate" | "near-term" | "mid-arc" | "slow-burn" | "endgame",
    notes: string                     // Additional context (default "")
  }>,
  chapterSummary?: {
    chapter: number,                  // Must match delta.chapter
    title: string,                    // Chapter title (required)
    characters: string,               // Characters who appeared
    events: string,                   // Key events
    stateChanges: string,             // State transitions
    hookActivity: string,             // Hook movements
    mood: string,                     // Emotional tone
    chapterType: string               // Chapter category
  },
  subplotOps: Array<Record<string, unknown>>,      // Subplot operations (loose schema)
  emotionalArcOps: Array<Record<string, unknown>>, // Emotional arc ops (loose schema)
  characterMatrixOps: Array<Record<string, unknown>>, // Character ops (loose schema)
  notes: string[]                     // Additional notes
}
```

### Hook Operation Semantics

- **upsert**: Create new hook or update existing (full HookRecord required)
- **mention**: Reference hook without changing status (keeps it alive)
- **resolve**: Mark hook as resolved (status → "resolved")
- **defer**: Mark hook as deferred (status → "deferred")

### Duplicate Hook Handling

When `upsert` targets an existing hook with same family:
- Merge `expectedPayoff` and `notes` (prefer richer text)
- Update `lastAdvancedChapter` to current chapter
- Set status to `progressing` if advanced
- Keep earliest `startChapter`

### Validation Rules

- `chapter` must be positive integer
- `chapterSummary.chapter` must match `delta.chapter` (when present)
- Hook IDs in `mention`, `resolve`, `defer` must be non-empty strings
- `upsert` hooks must pass `HookRecordSchema` validation
- No duplicate hook IDs within `upsert` array

### Logical Contradiction Detection

Truth file persistence validates delta for contradictions:
- Character cannot be in two locations simultaneously
- Resource delta cannot violate conservation laws
- Timeline cannot have backwards causality

When contradiction detected:
- Flag as critical audit issue
- Do NOT apply conflicting update
- Log contradiction in audit report

## 5. chapter-trace.json (chapter-XXXX.trace.json)

Optional debugging artifact tracking pipeline decisions.

### Schema

```typescript
{
  chapter: number,                    // Chapter number (≥1)
  preparerInputs: string[],            // Inputs used by Preparer
  selectedSources: string[],          // Context sources selected
  notes: string[]                     // Debugging notes
}
```

### Usage

Trace files are generated when `--trace` flag is enabled. They help debug:
- Why certain context was selected
- Which rules were active
- What inputs influenced planning decisions

## Quick Reference

### File Naming Convention

- Intent: `chapter-0001.intent.md` (markdown for readability)
- Context: `chapter-0001.context.json`
- Rule stack: `chapter-0001.rule-stack.yaml` (YAML for readability)
- Delta: `chapter-0001.delta.json`
- Trace: `chapter-0001.trace.json`

### Validation Entry Points

- `ChapterIntentSchema.parse()`: Validate intent
- `ContextPackageSchema.parse()`: Validate context package
- `RuleStackSchema.parse()`: Validate rule stack
- `RuntimeStateDeltaSchema.parse()`: Validate delta
- `ChapterTraceSchema.parse()`: Validate trace

### Common Errors

- `delta chapter goes backwards`: Delta chapter ≤ lastAppliedChapter
- `chapter summary mismatch`: Summary chapter ≠ delta chapter
- `duplicate summary row`: Chapter summary already exists
- `runtime state delta is not valid JSON`: JSON parse failure
- `runtime state delta failed schema validation`: Schema validation failure
